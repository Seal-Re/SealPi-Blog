## 阶段 A：身份体系与导航收口

### A-1 统一登录入口与用户头像展示

- **目标**
  - 全站只保留一套 GitHub 登录入口，登录后使用 GitHub 头像展示用户身份。

- **前置条件**
  - 现有 `auth.ts` GitHub 登录流程可正常工作。
  - `.env` 中 GitHub OAuth 配置正确。

- **实施步骤**
  1. 设计并实现统一的 `UserMenu` 组件（右上角头像/登录按钮），在所有页面复用。
  2. 在 `auth.ts` 的 `session` 回调中确保 `session.user` 包含：
     - `name`、`email`（如有）、`image`（GitHub 头像 URL）；
     - 后续会加入的 `githubUserId`、`role`。
  3. 在主布局（如 `app/layout.tsx` 或头部 `Header`）中仅使用 `UserMenu`，移除其他零散登录入口。
  4. 统一 `/admin/login` 行为：
     - 直接重定向到主登录 UI；或
     - 共享同一套 UI，仅作为别名路由。

- **输出物**
  - `UserMenu` 组件及统一登录 UI。

- **验收标准**
  - 任意页面右上角状态一致：
    - 未登录：仅显示“登录”按钮。
    - 已登录：显示 GitHub 头像和下拉菜单。
  - 代码中不再存在第二套登录入口。

- **风险 / 约束**
  - 避免在多处直接调用 `signIn`/`signOut`，统一包在 `UserMenu` 内。
  - SSR 场景下要避免 `auth()`/`useSession()` 带来的 UI 闪烁和 hydration 警告。

---

### A-2 引入并规范化角色字段（Role）

- **目标**
  - 在会话中显式引入 `role` 字段，区分普通用户 / 内容管理员 / 系统管理员。

- **前置条件**
  - A-1 登录流程稳定。

- **实施步骤**
  1. 约定首版角色：
     - `USER`：普通登录用户（仅浏览/评论）。
     - `CONTENT_ADMIN`：文章 CRUD 权限。
     - `SYS_ADMIN`：系统管控与监控权限。
  2. 在 `auth.ts` 的 `jwt` 回调中，根据：
     - 是否在 `ADMIN_GITHUB_IDS` 白名单；
     - 后端返回信息（如未来引入），
     为 `token.role` 赋值。
  3. 在 `session` 回调中，将 `token.role` 透传到 `session.user.role`。
  4. 在 `types/next-auth.d.ts` 中为 `Session` 与 `User` 类型补充 `role` 字段。

- **输出物**
  - 更新后的 `auth.ts` 和类型定义。
  - 一小节“角色定义说明”，可以写入本文档或 `implementation.md` 的权限章节。

- **验收标准**
  - 使用白名单账号和普通账号登录时，`session.user.role` 分别为 `CONTENT_ADMIN` 和 `USER`（或 `undefined`）等符合设计值。

- **风险 / 约束**
  - 首版角色不要过多，避免早期就引入复杂的权限矩阵。

---

### A-3 Admin 入口条件渲染（隐藏 admin tag）

- **目标**
  - 不再将 `admin` 以 tag 或显式入口形式直接暴露，改为基于登录状态与角色进行条件渲染。

- **前置条件**
  - A-2 中 `role` 字段可用。

- **实施步骤**
  1. 全局搜索项目中 `admin` 作为 tag 或导航项的出现位置。
  2. 对所有 Admin 入口进行条件渲染控制：
     - `role === 'CONTENT_ADMIN'`：显示“写文章”“文章管理”等。
     - `role === 'SYS_ADMIN'`：显示“系统管控”等。
  3. 从标签/文章元信息中剔除 `admin` 作为普通 tag 的展示。
  4. 明确定义“隐式入口”：
     - 例如头像下拉菜单中的“后台管理”，文章卡片上的“编辑”按钮，仅对 Admin 可见。

- **输出物**
  - 更新后的导航栏与相关入口渲染逻辑。

- **验收标准**
  - 未登录 & 普通用户访问站点时，页面 DOM 中不出现任何后台相关链接。
  - Admin 用户正常看到后台入口并可访问。

- **风险 / 约束**
  - SSR 和静态生成的页面中也必须不输出 Admin 入口 DOM，防止通过源码/爬虫看到隐式入口。

---

### A-4 `/admin` 路由保护与错误页体验

- **目标**
  - 从路由层明确区分“未登录”和“已登录但无权限”的体验。

- **前置条件**
  - A-1 ~ A-3 完成。

- **实施步骤**
  1. 审查 / 改造 `middleware.ts`：
     - 未登录访问 `/admin/**`：重定向到登录页。
     - 已登录但无 Admin 权限：跳转/渲染 403 页面。
  2. 实现通用的 `403` 页面（例如 `/forbidden`），包含清晰文案和“返回首页”按钮。
  3. 确保 `/admin/login` 本身不被中间件错误拦截，避免循环重定向。

- **输出物**
  - 更新后的 `middleware.ts` 和 `403` 页面。

- **验收标准**
  - 三种身份访问 `/admin`：
    - 未登录：进入登录页。
    - 已登录非 Admin：看到“无权限”。
    - Admin：直接进入后台首页。

- **风险 / 约束**
  - 要处理好 `next-auth` 的 `auth()`/`getServerSession()` 与中间件结合，避免性能与复杂度问题。

## 阶段 B：内容 CRUD、评论与标签

### B-1 文章 CRUD — 前后端契约校准

- **目标**
  - 确保 Admin 编辑器与后端文章管理接口在字段和行为上完全对齐。

- **前置条件**
  - A 阶段完成，管理接口已经可以被安全调用。

- **实施步骤**
  1. 对照 `docs/管理写接口与鉴权.md` + `ArticleAdminController`：
     - 整理所有写接口（URL、HTTP 方法、Query：`action=draft|publish`）。
     - 整理 multipart 字段（`title`、`url`、`draftJson`、`summary`、`previewImage` 等）。
  2. 在前端封装统一 Admin API Client（如 `lib/admin-api.ts`）：
     - `saveDraft` / `publishArticle` / `updateDraft` / `updateAndPublish` 等语义化方法。
  3. `AdminEditorClient` 只依赖 API Client，不直接使用 `fetch`。
  4. 本地验证草稿/发布行为，确认 `draftJson` / `contentJson` 行为与设计一致。

- **输出物**
  - Admin API Client 及“接口字段对照表”。

- **验收标准**
  - 跑通流程：创建草稿 → 发布 → 修改草稿 → 再次发布，数据库字段变化符合设计文档。

- **风险 / 约束**
  - 若仍需兼容 JSON 版接口，应在代码与文档中标记 `@Deprecated` 和废弃计划，而不是直接删除。

---

### B-2 Excalidraw 图片外置与 JSON 优化【DONE】

- **目标**
  - 将 Excalidraw 图片从 Base64 内联改为 OSS URL 引用，控制 JSON 体积。

- **前置条件**
  - `/api/v1/admin/upload` 和 MinIO/OSS 已能正常工作。

- **实施步骤**
  1. 在 `AdminEditorClient` 中集成 Excalidraw 的图片输入 Hook：
     - 处理粘贴、拖拽、选择文件事件。
  2. 对每次图片输入：
     - 调用 Admin API Client 的 `uploadImage` 上传，获得 URL。
     - 修改 Excalidraw JSON，使图片使用 URL 引用而非 Base64。
  3. 在保存草稿/发布前，可选增加简单检测：
     - 若 JSON 中出现超大 Base64 字符串，提示用户或阻止提交。

- **输出物**
  - 更新后的 `AdminEditorClient` + `uploadImage` 调用链。

- **验收标准**
  - 新文章的 `draftJson` / `contentJson` 中不再出现大段 Base64，单篇 JSON 体积明显减小。

- **风险 / 约束**
  - 需兼容既有 Base64 数据的展示，迁移老数据可后续通过离线脚本完成。

---

### B-3 列表 / 首页 / 详情接口统一（动态内容源）【DONE】

- **目标**
  - 前台 `/blog` & 首页完全使用后端公共读接口，而不是本地 MDX。

- **前置条件**
  - 后端公共读接口（分页列表 / 详情 / 按 slug）已实现并稳定。

- **实施步骤**
  1. 完成 `public-blog-api.ts`：
     - 如 `getArticles(page)`、`getArticleBySlug(slug)`、`getTags()`。
  2. 改造前台页面：
     - `app/page.tsx`：通过公共读接口拉取首页展示数据。
     - `app/blog/page.tsx`、`app/blog/page/[page]/page.tsx`：分页列表使用 API 数据。
     - `app/blog/[...slug]/page.tsx`：
       - 调用 `getArticleBySlug`；
       - 使用 `<ExcalidrawViewer />` 渲染 `contentJson`；
       - 使用 `coverImageUrl` 作为头图和 Open Graph 图。
  3. 将 Contentlayer/MDX 仅保留为非主业务用途（如静态文档页），并在代码中剥离其与 /blog 主链路的耦合。

- **输出物**
  - 更新后的 `/blog` 系列页面与 API Client。

- **验收标准**
  - 删除 / 改名 `data/blog/*.mdx` 文件后，前台 `/blog` 与文章详情仍能正常工作。
  - 后端关闭时，前台能给出清晰的错误提示，而不是白屏。

- **风险 / 约束**
  - 需重新评估 SSR / SSG 策略，保证 SEO 不因完全动态化而明显退化。

---

### B-4 标签体系动态化【DONE】

- **目标**
  - 标签列表与按标签过滤完全使用后端数据，废弃静态 `tag-data.json`。

- **前置条件**
  - 后端 Tag 相关读接口到位。

- **实施步骤**
  1. 明确 Tag 接口契约：
     - `GET /api/v1/tags`：返回标签列表 + 文章数量。
     - `GET /api/v1/articles?tag=xxx&page=n`：按标签分页。
  2. 改造前端：
     - `app/tags/page.tsx`：使用 `GET /tags` 渲染标签列表。
     - `app/tags/[tag]/page.tsx` 和 `app/tags/[tag]/page/[page]/page.tsx`：按标签调用分页接口。
  3. `tag-data.json` 标记为 deprecated 或删除（如果不再需要）。

- **输出物**
  - 更新后的标签页面与 Tag API 调用。

- **验收标准**
  - 增删文章标签后，前端标签页面展示与后端数据一致。

- **风险 / 约束**
  - 需规范标签命名规则（大小写、URL encode 等），避免多种写法对应同一标签。

---

### B-5 评论系统重构（GitHub 账号）

- **目标**
  - 废弃邮箱注册评论，统一使用 GitHub 账号进行评论。

- **前置条件**
  - 阶段 A 身份体系已稳定。

- **实施步骤**
  1. 梳理现有评论模块：
     - 找出使用邮箱注册/登录的代码与接口。
     - 评估老评论数据是否需要展示。
  2. 设计新评论数据结构与接口：
     - 评论字段：`articleId/slug`、`authorGithubId`、`authorName`、`authorAvatar`、`content`、`createdAt`。
     - API：
       - `GET /api/v1/articles/{id}/comments`
       - `POST /api/v1/articles/{id}/comments`（需登录）。
  3. 前端实现：
     - 详情页中：
       - 已登录：显示评论输入框和 GitHub 头像。
       - 未登录：显示“使用 GitHub 登录后发表评论”的提示。
  4. 清理邮箱注册/评论相关 UI 与接口调用。

- **输出物**
  - 评论 API 与前端评论区域。

- **验收标准**
  - 未登录用户无法发表评论。
  - 登录用户发表评论时，前端显示的头像和名称来自 GitHub。

- **风险 / 约束**
  - 如旧评论需要保留展示，需要单独设计只读兼容策略。

## 阶段 C：后台运维与监控（系统管控）

### C-1 Admin 信息架构拆分（内容 / 运维）

- **目标**
  - 从信息架构上把 “内容管理” 与 “系统运维” 两个域拆开，减少未来耦合。

- **前置条件**
  - Admin 入口与角色控制已完成。

- **实施步骤**
  1. 设计 `/admin` 子导航结构：
     - `/admin/content`：文章列表、标签管理、草稿箱。
     - `/admin/ops`：系统监控面板（首版）。
  2. 调整现有 `/admin/page.tsx` 为后台首页，展示两个入口卡片。
  3. 基于 `role` 控制可见性：
     - `CONTENT_ADMIN`：仅能访问内容管理。
     - `SYS_ADMIN`：可访问运维域。

- **输出物**
  - 更新后的 `/admin` 导航与首页。

- **验收标准**
  - 不同角色访问后台首页时，仅能看到自己权限范围内的入口。

- **风险 / 约束**
  - 路由规划要留出扩展空间，避免未来大规模 URL 重构。

---

### C-2 基础系统监控面板（首版）

- **目标**
  - 为系统管理员提供基础运行状态视图。

- **前置条件**
  - C-1 路由结构已定。

- **实施步骤**
  1. 确定首版展示指标：
     - 应用版本、启动时间、数据库连通性、磁盘使用大致情况等。
  2. 在后端提供只读监控接口：
     - 可结合 Spring Boot Actuator 做简单封装，如 `/api/v1/admin/ops/status`。
  3. `/admin/ops` 页面消费该接口，用卡片/表格展示状态。

- **输出物**
  - 监控接口与 `/admin/ops` 页面。

- **验收标准**
  - 在正常与异常两种状态下，页面均有明确展示（例如数据库不可用时有红色告警块）。

- **风险 / 约束**
  - 接口中避免返回敏感信息（如完整连接串、密码等）。

## 覆盖性与风险检查总结

- **核心用户流程**
  - 浏览 → 登录 → 评论；
  - 管理员登录 → 文章草稿/发布 → 前端展示；
  - 系统管理员登录 → 查看运维面板。
- **非功能性工作**
  - 测试（D-1）、部署（D-2）、监控（C-2/D-3）、文档与 README（D-4）均有覆盖。
- **风险与边界处理**
  - 会话缺失 / 角色缺失：通过安全兜底（不渲染敏感入口）。
  - 网络失败 / 写接口失败：Admin 侧需要良好错误提示与可选重试。
  - 历史数据（Base64 图片、旧评论）：通过兼容展示与后续迁移策略处理。
